Queries — Error Handling

Sometimes errors happen! There are a few ways to express them in GraphQL:

  • Expose errors in the schema itself
  • Add errors to the response’s "errors" key

Additionally, you can provide opt-in error support by wrapping resolvers (described below).

Exposing Errors

You can expose errors as part of your schema. In a sense, this makes errors part of “business as usual”, instead of exceptional cases.

For example, you could add errors to PostType:

1PostType = GraphQL::ObjectType.define do
2  name "Post"
3  field :title, types.String
4  # ...
5  field :errors, types[types.String], "Reasons the object couldn't be created or updated" do
6    resolve ->(obj, args, ctx) { obj.errors.full_messages }
7  end
8end

Then, when creating a post, return the Post, even if the save failed:

1resolve ->(obj, args, ctx) {
2  post = Post.new(args["post"].to_h)
3  # Maybe this fails, no big deal:
4  post.save
5  post
6}

Then, when clients create a post, they should check the errors field to see if it was successful:

1mutation {
2  createPost(post: {title: "GraphQL is Nice"}) {
3    id
4    title
5    errors # in case the save failed
6  }
7}

If errors is present (and id is null), the client knows that the operation was unsuccessful, and they can discover why.

This technique could be extended by creating dedicated error types, too.

The “errors” Key

A GraphQL response may have an "errors" key, for example:

1Schema.execute(query_string)
2# {
3#   "errors" => [
4#     { "message" => "Something went wrong" },
5#   ],
6# }

You can add an error to the "errors" key by returning a GraphQL::ExecutionError from a resolve function. For example:

 1resolve ->(obj, args, ctx) {
 2  post_params = args["post"].to_h
 3  if obj.posts.create(post_params)
 4    # on success, return the post:
 5    post
 6  else
 7    # on error, return an error:
 8    GraphQL::ExecutionError.new("Invalid input for Post: #{post.errors.full_messages.join(", ")}")
 9  end
10}

If some part of your resolve function would raise an error, you can rescue it and return a GraphQL::ExecutionError instead:

 1resolve ->(obj, args, ctx) {
 2  post_params = args["post"].to_h
 3  begin
 4    post = obj.posts.create!(post_params)
 5    # on success, return the post:
 6    post
 7  rescue ActiveRecord::RecordInvalid => err
 8    # on error, return an error:
 9    GraphQL::ExecutionError.new("Invalid input for Post: #{post.errors.full_messages.join(", ")}")
10  end
11}

Error Handling with Wrappers

If you don’t want to begin ... rescue ... end in every field, you can wrap resolve functions in error handling. For example, you could make an object that wraps another resolver:

 1# Wrap field resolver `resolve_func` with a handler for `error_superclass`.
 2# `RescueFrom` instances are valid field resolvers too.
 3class RescueFrom
 4  def initialize(error_superclass, resolve_func)
 5    @error_superclass = error_superclass
 6    @resolve_func = resolve_func
 7  end
 8
 9  def call(obj, args, ctx)
10    @resolve_func.call(obj, args, ctx)
11  rescue @error_superclass => err
12    # Your error handling logic here:
13    # - return an instance of `GraphQL::ExecutionError`
14    # - or, return nil:
15    nil
16  end
17end

Then, apply it to fields on an opt-in basis:

1field :create_post, PostType do
2  # Wrap the resolve function with `RescueFrom.new(err_class, ...)`
3  resolve RescueFrom.new(ActiveRecord::RecordInvalid, ->(obj, args, ctx) { ... })
4end

This way, you get error handling with proper Ruby semantics and no overhead!
