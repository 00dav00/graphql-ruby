Relay — Connections

Relay expresses one-to-many relationships with connections. Connections support pagination, filtering and metadata in a robust way.

graphql-ruby includes built-in connection support for Array, ActiveRecord::Relations, and Sequel::Datasets. You can define custom connection classes to expose other collections with GraphQL.

Connection fields

To define a connection field, use the connection helper. For a return type, get a type’s .connection_type. The resolve proc should return a collection (eg, Array or ActiveRecord::Relation) without pagination. (The connection will paginate the collection).

For example:

 1PostType = GraphQL::ObjectType.define do
 2  # `Post#comments` returns an ActiveRecord::Relation
 3  # The GraphQL field returns a Connection
 4  connection :comments, CommentType.connection_type
 5  # `Post#similar_posts` returns an Array
 6  connection :similarPosts, PostType.connection_type, property: :similar_posts
 7
 8  # ...
 9end

You can also define custom arguments and a custom resolve function for connections, just like other fields:

 1connection :featured_comments, CommentType.connection_type do
 2  # Add an argument:
 3  argument :since, types.String
 4
 5  # Return an Array or ActiveRecord::Relation
 6  resolve ->(post, args, ctx) {
 7    comments = post.comments.featured
 8
 9    if args[:since]
10      comments = comments.where("created_at >= ", since)
11    end
12
13    comments
14  }
15end

Maximum Page Size

You can limit the number of results with max_page_size::

1connection :featured_comments, CommentType.connection_type, max_page_size: 50

Connection types

You can customize a connection type with .define_connection:

 1# Make a customized connection type
 2PostConnectionWithTotalCountType = PostType.define_connection do
 3  name "PostConnectionWithTotalCount"
 4  field :totalCount do
 5    type types.Int
 6    # - `obj` is the Connection
 7    # - `obj.nodes` is the collection of Posts
 8    resolve ->(obj, args, ctx) { obj.nodes.count }
 9  end
10end
11

Now, you can use PostConnectionWithTotalCountType to define a connection with the “totalCount” field:

1AuthorType = GraphQL::ObjectType.define do
2  # Use the custom connection type:
3  connection :posts, PostConnectionWithTotalCountType
4end

This way, you can query your custom fields, for example:

1{
2  author(id: 1) {
3    posts {
4      totalCount    # <= Your custom field
5    }
6  }
7}

Custom edge types

If you need custom fields on edges, you can define an edge type and pass it to a connection:

 1# Person => Membership => Team
 2MembershipSinceEdgeType = TeamType.define_edge do
 3  name "MembershipSinceEdge"
 4  field :memberSince, types.Int, "The date that this person joined this team" do
 5    resolve ->(obj, args, ctx) {
 6      obj # => GraphQL::Relay::Edge instance
 7      person = obj.parent
 8      team = obj.node
 9      membership = Membership.where(person: person, team: team).first
10      membership.created_at.to_i
11    }
12  end
13end

Then, pass the edge type when defining the connection type:

1TeamMembershipsConnectionType = TeamType.define_connection(edge_type: MembershipSinceEdgeType) do
2  # Use a name so it doesn't conflict with "TeamConnection"
3  name "TeamMembershipsConnection"
4end

Now, you can query custom fields on the edge:

 1{
 2  me {
 3    teams {
 4      edge {
 5        memberSince     # <= Here's your custom field
 6        node {
 7          teamName: name
 8        }
 9      }
10    }
11  }
12}

Custom Edge classes

For more robust custom edges, you can define a custom edge class. It will be obj in the edge type’s resolve function. For example, to define a membership edge:

 1# Make sure to familiarize yourself with GraphQL::Relay::Edge --
 2# you have to avoid naming conflicts here!
 3class MembershipSinceEdge < GraphQL::Relay::Edge
 4  # Cache `membership` to avoid multiple DB queries
 5  def membership
 6    @membership ||= begin
 7      # "parent" and "node" are passed in from the surrounding Connection,
 8      # See `Edge#initialize` for details
 9      person = self.parent
10      team = self.node
11      Membership.where(person: person, team: team).first
12    end
13  end
14
15  def member_since
16    membership.created_at.to_i
17  end
18
19  def leader?
20    membership.leader?
21  end
22end

Then, hook it up with custom edge type and custom connection type:

 1# Person => Membership => Team
 2MembershipSinceEdgeType = BaseType.define_edge do
 3  name "MembershipSinceEdge"
 4  field :memberSince, types.Int, "The date that this person joined this team", property: :member_since
 5  field :isPrimary, types.Boolean, "Is this person the team leader?", property: :primary?
 6end
 7
 8TeamMembershipsConnectionType = TeamType.define_connection(
 9    edge_class: MembershipSinceEdge,
10    edge_type: MembershipSinceEdgeType,
11  ) do
12  # Use a name so it doesn't conflict with "TeamConnection"
13  name "TeamMembershipsConnection"
14end

Connection objects

Maybe you need to make a connection object yourself (for example, to return a connection type from a mutation). You can create a connection object like this:

1items = [...]     # your collection objects
2args = {}         # stub out arguments for this connection object
3connection_class = GraphQL::Relay::BaseConnection.connection_for_nodes(items)
4connection_class.new(items, args)

.connection_for_nodes will return RelationConnection or ArrayConnection depending on items, then you can make a new connection

For specifying a connection based on an ActiveRecord::Relation or Sequel::Dataset:

1object = {}       # your newly created object
2items = [...]     # your AR or Sequel collection
3args = {}         # stub out arguments for this connection object
4items_connection = GraphQL::Relay::RelationConnection.new(
5  items,
6  args
7)
8edge = GraphQL::Relay::Edge.new(object, items_connection)

Additionally, connections may be provided with the GraphQL::Field that created them. This may be used for custom introspection or instrumentation. For example,

1  Schema.get_field(TodoListType, "todos")
2  # => #<GraphQL::Field name="todos">
3  context.irep_node.definitions[TodoListType]
4  # => #<GraphQL::Field name="todos">
5  # although this one may not work with fields on interfaces

Custom connections

You can define a custom connection class and add it to GraphQL::Relay.

First, define the custom connection:

 1require "set" # From Ruby's standard library
 2class SetConnection < BaseConnection
 3  # derive a cursor from `item`
 4  def cursor_from_node(item)
 5    # ...
 6  end
 7
 8  private
 9  # apply `#first` & `#last` to limit results
10  def paged_nodes
11    # ...
12  end
13
14  # apply cursor, order, filters, etc
15  # to get a subset of matching objects
16  def sliced_nodes
17    # ...
18  end
19end

Then, register the new connection with GraphQL::Relay::BaseConnection:

1# When exposing a `Set`, use `SetConnection`:
2GraphQL::Relay::BaseConnection.register_connection_implementation(Set, SetConnection)

At runtime, GraphQL::Relay will use SetConnection to expose Sets.

Creating connection fields by hand

If you need lower-level access to Connection fields, you can create them programmatically. Given a GraphQL::Field which returns a collection of items, you can turn it into a connection field with ConnectionField.create.

For example, to wrap a field with a connection field:

1field = GraphQL::Field.new
2# ... define the field
3connection_field = GraphQL::Relay::ConnectionField.create(field)
