Schema — Types and Fields

Types, fields and arguments make up a schema’s type system. These objects are also open to extension via metadata and accepts_definitions.

Types

Types describe objects and values in a system. The API documentation for each type contains a detailed description with examples.

Objects are described with GraphQL::ObjectTypes.

Scalar values are described with built-in scalars (string, int, float, boolean, ID) or custom GraphQL::EnumTypes. You can define custom GraphQL::ScalarTypes, too.

Scalars and enums can be sent to GraphQL as inputs. For complex inputs (key-value pairs), use GraphQL::InputObjectType.

There are two abstract types, too:

  • GraphQL::InterfaceType describes a collection of object types which implement some of the same fields.
  • GraphQL::UnionType describes a collection of object types which may appear in the same place in the schema (ie, may be returned by the same field.)

GraphQL::ListType and GraphQL::NonNullType modify other types, describing them as “list of T” or “required T”.

Referencing Types

Some parts of schema definition take types as an input. There are two good ways to provide types:

  1. By value. Pass a variable which holds the type.

    1# constant
2field :team, TeamType
3# local variable
4field :stadium, stadium_type

  2. By proc, which will be lazy-evaluated to look up a type.

    1field :team, -> { TeamType }
2field :stadium, -> { LookupTypeForModel.lookup(Stadium) }

Fields

GraphQL::ObjectTypes and GraphQL::InterfaceTypes may expose their values with fields. A field definition looks like this:

1PostType = GraphQL::ObjectType.define do
2  # ...
3  #     name  , type        , description (optional)
4  field :title, types.String, "The title of the Post"
5end

By default, fields are resolved by sending the name to the underlying object (eg post.title in the example above).

You can define a different resolution by providing a resolve function:

 1PostType = GraphQL::ObjectType.define do
 2  # ...
 3  #     name   , type        , description (optional)
 4  field :teaser, types.String, "The teaser of the Post" do
 5    # how to get the value?
 6    resolve ->(obj, args, ctx) {
 7      # first 40 chars of the body
 8      obj.body[0, 40]
 9    }
10  end
11end

The resolve function receives inputs:

  • object: The underlying object for this type (above, a Post instance)
  • arguments: The arguments for this field (see below, a GraphQL::Query::Arguments instance)
  • context: The context for this query (see “Executing Queries”, a GraphQL::Query::Context instance)

In fact, the field do ... end block is passed to GraphQL::Field’s .define method, so you can define many things there:

1field do
2  name "teaser"
3  type types.String
4  description "..."
5  resolve ->(obj, args, ctx) { ... }
6  deprecation_reason "Too long, use .title instead"
7  complexity 2
8end

Arguments

Fields can take arguments as input. These can be used to determine the return value (eg, filtering search results) or to modify the application state (eg, updating the database in MutationType).

Arguments are defined with the argument helper:

 1field :search_posts, types[PostType] do
 2  argument :category, types.String
 3  resolve ->(obj, args, ctx) {
 4    args[:category]
 5    # => maybe a string, eg "Programming"
 6    if args[:category]
 7      Post.where(category: category).limit(10)
 8    else
 9      Post.all.limit(10)
10    end
11  }
12end

Use ! to mark an argument as required:

1# This argument is a required string:
2argument :category, !types.String

Only certain types are valid for arguments:

  • GraphQL::ScalarType, including built-in scalars (string, int, float, boolean, ID)
  • GraphQL::EnumType
  • GraphQL::InputObjectType, which allows key-value pairs as input
  • GraphQL::ListTypes of a valid input type
  • GraphQL::NonNullTypes of a valid input type

The args parameter of a resolve function will always be a <a href="http://www.rubydoc.info/gems/graphql/GraphQL/Query/Arguments" target="_blank"><code>GraphQL::Query::Arguments</code></a>. You can access specific arguments with ["arg_name"] or [:arg_name]. You recursively turn it into a Ruby Hash with to_h. Inside args, scalars will be parsed into Ruby values and enums will be converted to their value: (if one was provided).

1resolve ->(obj, args, ctx) {
2  args["category"] == args[:category]
3  # => true
4  args.to_h
5  # => { "category" => "Programming" }
6  # ...
7}

Extending type and field definitions

Types, fields, and arguments have a metadata hash which accepts values during definition.

First, make a custom definition:

1GraphQL::ObjectType.accepts_definitions resolves_to_class_names: GraphQL::Define.assign_metadata_key(:resolves_to_class_names)
2# or:
3# GraphQL::Field.accepts_definitions(...)
4# GraphQL::Argument.accepts_definitions(...)

Then, use the custom definition:

1Post = GraphQL::ObjectType.define do
2  # ...
3  resolves_to_class_names ["Post", "StaffUpdate"]
4end

Access type.metadata later:

 1MySchema = GraphQL::Schema.define do
 2  # ...
 3  # Use the type's declared `resolves_to_class_names`
 4  # to figure out if `obj` is a member of that type
 5  resolve_type ->(obj, ctx) {
 6    class_name = obj.class.name
 7    MySchema.types.values.find { |type| type.metadata[:resolves_to_class_names].include?(class_name) }
 8  }
 9end

This behavior is provided by GraphQL::Define::InstanceDefinable.
